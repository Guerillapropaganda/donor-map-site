## Page-by-page findings

### /signoff-queue trace — the big lie

**Data flow:**
1. `/signoff-queue` page reads `/api/vault` which parses every profile.
2. Page filters profiles where `auditAPlusPassed` is truthy AND `lastVerifiedBy !== "editorial"`.
3. `auditAPlusPassed` is frontmatter field `audit-a-plus-passed`, stamped by `scripts/pipeline-janitor.cjs --tier=a-plus --cohort --write`.

**The lie (P-007 BLOCKING):** the stamp means completely different things for different profile types.

The janitor's A+ quality checks (`pipeline-janitor.cjs:284-351`) are gated on `type === 'politician'`:
- committee-relevant regulatory cross-ref
- 3+ Tier 1 source types
- automated legal-review pass (defamation-prone phrases outside blockquotes)
- both-sides entity detection
- `central-thesis` populated
- `story-grade` populated

For donors, corporations, and everything else, NONE of those checks run.

For non-politicians, the stamp fires when just three trivially-weak conditions pass:
- no "zombie" auto-blocks
- no "missing" auto-blocks
- `last-enriched` < 90 days old
- `content-readiness: ready`

**Observed reality of the queue:**
- 125 profiles stamped total
- 72 donors
- 50 corporations
- 3 "other" (2 sub-notes, 1 unknown)
- **0 politician master profiles**
- 413 politician master profiles are `content-readiness: ready` but unstamped — because they actually face real quality checks and fail

**What the UI tells David:** "125 profiles passed every A+ check and are waiting on your editorial sign-off."

**Actual meaning:** "122 non-politician profiles passed a very weak check; 0 politicians passed the real check; you've been staring at a queue of non-politician profiles while the politician profiles — the ones with the highest defamation risk and editorial complexity — aren't represented at all."

This is THE pattern David described: checklist reports things as "ready" that aren't actually verified.

### P-008 (STALE vocab): "S-tier" in signoff-queue UI
`ops/src/app/signoff-queue/page.tsx:85-86, 128-131, 260-266, 294-296` references S-tier
promotion. ADR-0017 (2026-04-23) rewrote the readiness flow as
`raw → draft → ready → data-complete → verified`. S-tier is not in the current
vocabulary. The page's "S-tier ready" column + footer copy are drift from the
current tier system.

### P-009 (STALE logic): "pipeline-janitor --tier=a-plus" empty-state instructions
Same page at line 192-195 tells David to run:
`node scripts/pipeline-janitor.cjs --tier=a-plus --cohort --write`
But per ADR-0017 the new readiness pipeline is `scripts/reclassify-readiness.cjs`.
This empty-state instruction may be pointing at obsolete tooling.

### P-010 (BLOCKING): janitor's `audit-a-plus-passed` stamp is one-way
Once stamped, the field is never cleared automatically. If a profile later
develops problems (prose-data mismatch, URL rot, source downgrade), the stamp
stays. The only way off the queue is editorial sign-off. A profile that was
A+ in February can be broken today and still look "waiting on David's sign-off."

### P-011 (BLOCKING): the janitor doesn't check prose-vs-structured-data mismatches
David: "Currently we keep finding mistakes with the numbers" — prose says
one dollar amount, auto-block graph says another. The janitor checks:
- Is the auto-block present? ✓
- Is the frontmatter field set? ✓
- Does the body sum/restate match the auto-block data? ✗ (not checked)

So a profile saying "ADM gave $2.4M" in prose while its auto-block shows
$1.8M passes A+ today. This is a gap in the checker, not a bug in the stamp.

## Verdict on sign-off queue

The queue isn't wrong about WHO is on it — those 125 profiles do have the
stamp. But the stamp doesn't mean what the UI claims it means. The mental model
"passed every A+ check → just needs sign-off" is false for 122 of 125 entries.

**Fix shape (for later discussion):**
1. Remove the stamp for non-politicians OR build A+ checks for donors/corporations too.
2. Fix the UI copy + readiness-tier vocabulary to match ADR-0017.
3. Add the prose-vs-structured-data check that catches the "numbers don't match" class of bugs.
4. Make the stamp reversible — nightly re-audit that clears stamps on profiles that drifted.

### /signoff-launch trace — same disease, different body

**Data flow:**
1. `/signoff-launch` page reads `/api/launch-50`.
2. API reads `content/Admin Notes/launch-50-audit.json` (audit) + `ops/data/launch-50-signoff.json` (user checkbox state).
3. Audit is generated by running `node scripts/launch-50-audit.cjs` manually.
4. Page shows 4-column progress bar: Class Analysis, URLs Verified, David Reviewed, Verified.

**P-012 (BLOCKING): audit JSON is a frozen snapshot, not a live check.**
The audit file captures `readiness`, `sourceCount`, `urlNeeded`, `unverified`, `needsReview`,
`hasClassAnalysis`, `autoBlocks` at the moment `launch-50-audit.cjs` runs. Nothing
auto-refreshes. Page displays whatever was true at last audit run, no matter
how old. No timestamp shown in the UI.

**Confirmed concrete lie:** audit JSON (generated today 2026-04-23 12:22 PM)
reports Kamala Harris as `readiness: "draft"`, `sourceCount: 0`. Actual current
frontmatter: `content-readiness: data-complete`, has `source-types: [Congress, ...]`.
The checklist is showing stale state. David looking at the page right now
would see Kamala blocking on things she's already past.

**P-013 (BLOCKING): 3 of 4 checkboxes are manual toggles stored in a gitignored file.**
`ops/data/launch-50-signoff.json` — per route.ts:7. Verified the directory doesn't
even exist yet (`ls ops/data/` returned "No such file or directory"). This means:
- Progress is lost on machine switch
- Progress not committed to git, so ANY backup/restore wipes it
- If the worktree is deleted, progress is gone
- No audit trail of who toggled what when (just an `updatedAt` timestamp)

**P-014 (BLOCKING): manual checkbox overrides audit data.**
`signoff-launch/page.tsx:103,111,147` — `hasCA = s.classAnalysisWritten || r.hasClassAnalysis`.
If the audit finds no Class Analysis heading, but David ticks the checkbox anyway,
the dashboard goes green. Nothing stops false-positive checkmarks. The "Class
Analysis" box was intended to be automatic (from audit) but made manually-overridable.

**P-015 (BLOCKING): status calculation ignores readiness tier.**
`signoff-launch/page.tsx:108-116` — a profile at `readiness: draft` can be marked
`ready-to-sign` if David ticks all the checkboxes. The status function only
checks: has CA + urls verified + no flags. Nothing stops a draft-tier profile
from reporting ready-to-sign. (Exception: `promoted` checkbox OR `readiness===verified`
→ 'done'. But the path to "ready-to-sign" doesn't block on readiness.)

**P-016 (BLOCKING): different source-count logic than sign-off queue.**
- sign-off queue's janitor (`pipeline-janitor.cjs:301`): `Math.max((data['source-types'] || []).length, countTier1InBody(body))`
- launch-50 audit (`launch-50-audit.cjs:114`): `(fm["source-types"] || []).length` — NO body fallback.

The two checklists count sources differently. A profile with Tier 1 source
refs in prose but empty frontmatter `source-types` would show zero sources
on launch-50, some sources on sign-off queue. Same profile, two different
truths, and no indication to the user which is right.

**P-017 (UX): hasClassAnalysis is just a heading regex.**
`launch-50-audit.cjs:113` — `/^##\s+Class Analysis/m.test(body)`. A profile
with an empty `## Class Analysis` heading passes. Short, useless content passes.
Doesn't verify quality, length, structured nature, or whether the required
sub-sections exist (per Class Analysis rubric in ADR-0001).

**P-018 (ORPHAN): launch-50-audit.cjs hardcodes the 50 names in source.**
`launch-50-audit.cjs:21-47` — the LAUNCH_50 constant is a hardcoded JS array.
Changing the 50 requires editing the script and running it again. The ops UI
has no way to add/remove profiles from the launch list.

**P-019 (UX): the launch-50 page is not in the sidebar.**
Duplicate of P-003 — reconfirmed during trace. Launch-50 signoff is the most
stake-having surface in the app and requires URL-typing to reach.

**P-020 (UX): findProfile() uses fuzzy match that can resolve to wrong file.**
`launch-50-audit.cjs:86-103` — if a name doesn't match exactly, falls back to
partial match with heuristics ("prefer Master Profile", "prefer non-sub-note").
For a name like "George Soros" with multiple matching files (a donor profile
and a children-funded-by page), it could silently pick the wrong one and
audit the wrong profile. No warning surfaced to the UI when this happens.

# ADR-0024 Prevention Checklist

How the donor-map graph engine and surrounding harness will keep the
class of bugs surfaced during the 2026-04-25 shadow-diff walk from
recurring. Built from the question David asked: *"how does this not
become a problem in the future?"*

## What we caught walking the diff (2026-04-25)

Three classes of bug, all visible in the first shadow-mode scan against
production:

1. **Pathless ghost stubs** — entity records like `ent_001546 "Bob Casey"`
   and `ent_001550 "Dan Goldman"`, created by `discovery-scanner` from
   wikilinks, with `profile_path: null`. They shadowed the real profiles
   stored under formal names (`Robert P. Casey`, `Daniel S. Goldman`).
2. **Duplicate profile folders for one human** — `Ed Markey/` AND
   `Edward J. Markey/`, `Jim Himes/` AND `James A. Himes/`. Two folders,
   two entity records, one politician. The librarian correctly flags
   these as ambiguous and refuses to resolve.
3. **Cache duplicate keys** — `relationships-per-profile.json` had
   separate keys for `"Kamala Harris"` and `"KAMALA HARRIS FOR SENATE"`.
   Same person, two cache entries. The librarian unifies them via
   FEC registry aliasing.

## How each layer prevents recurrence

### Layer 1 — Librarian as single read path (architectural)

**Status:** skeleton shipped 2026-04-25, consumer migration in flight.

Once `scripts/build-relationships-per-profile.cjs` and the rest of the
ops surfaces are migrated to call the librarian, the cache and ops
pages become one read path. The class-3 bug (cache duplicate keys)
becomes structurally impossible — the cache is *generated by* the
librarian, so it can't disagree with the librarian.

This is the biggest single payoff. Until Phase 3 lands, drift between
cache and live reads can keep forming.

**What's done:** `lib/donor-map/` library with `Graph.load()` →
`resolve / neighbors / aggregate`. Shadow-mode comparator on
`/api/profile/edges` for the `related` field.

**What's open:**
- Port `scripts/lib/edge-role-taxonomy.cjs` to TypeScript so the
  shadow can expand to `donors` and `politicians-funded`.
- Migrate the cache rebuilder to consume the librarian (Phase 3).
- Migrate remaining ops surfaces (`/relationships`, `/money-trail`,
  `/capitol-trades`, `/connections`, `/contradictions`, `/query`,
  `/ask`) — order by usage frequency.

### Layer 2 — Load-time validation (engine refuses bad data)

**Status:** live in `Resolver.build()`.

The librarian throws on the genuinely dangerous patterns at startup:

- `DuplicateBioguideError` — two profiles claiming the same bioguide id
  (defamation-adjacent: would put one politician's voting record on
  another's profile).
- `FecRegistryConflictError` — one FEC committee id mapped to multiple
  vault profiles (the Fairshake/Future-Forward-USA bug class).

Aliases that collide soft-fail into `ambiguous_aliases` instead of
throwing — engine still starts; queries against the ambiguous name
return `UnresolvableError` with a candidate list. This is the right
balance because corp-vs-PAC name overlap is legitimate (Apollo Global
Management and its PAC arm can share branding), but it does mean the
ambiguity has to be visible somewhere — see Layer 3.

### Layer 3 — Harness checks (catch new bad data when it's introduced)

**Status:** four checks proposed, none built yet. This is the gap.

The harness already runs every 15 minutes via `attention-dispatcher.cjs`
and surfaces findings to `/attention` and the Dashboard. The four
proposed checks plug into that same pipeline.

#### Check 1 — `librarian-validation-status` (META)

The single most important new check. Runs `Graph.load()` and reports:
- exit !== 0 if the engine throws (`DuplicateBioguideError`,
  `FecRegistryConflictError`, file-missing, etc.) — this turns into a
  red Dashboard chip and blocks merges if promoted to pre-commit.
- `findings_count = ambiguous_aliases.size` as a tracked number — turns
  yellow if it grows over time, indicating new alias collisions.
- `unresolved_edges` count as a secondary signal — currently 79K of
  152K edges (52%); if that climbs further, something is regressing
  upstream.

This is a "meta" check in that it doesn't enumerate specific bug
patterns — it makes the librarian's verdict on the data part of vault
health. Failure here means *something* is wrong; the engine output
points at what.

#### Check 2 — `pathless-stub-entities`

Scans `entities.jsonl` for records with `profile_path: null`. For each,
classifies whether it's a legitimate no-profile type (industry blocs,
aggregation nodes) or a likely ghost. Surfaces ghost candidates to
`/attention` for triage — David picks "create the profile" or "delete
the entity record."

Catch the next Bob Casey / Dan Goldman situation when the wikilink is
typed, not six months later.

#### Check 3 — `duplicate-politician-profiles`

Cross-references `legislator-registry.jsonl` with `entities.jsonl` to
flag any case where two distinct vault profiles map to the same
politician. Two signals:
- Same bioguide id present on two profiles (already caught by
  `duplicate-bioguide-sentinel` pre-commit, but background re-scan
  catches things that bypassed the sentinel).
- Same `name_first + name_last` across two politician-type entities
  with distinct profile paths.

Catches Ed Markey + Edward J. Markey when the second profile is
created. David triages: pick canonical, archive the other.

#### Check 4 — `unresolvable-edge-endpoints-trend`

Tracks `unresolved_edges` count over time. Today's baseline: 79,838.
Expected to drop as duplicate profiles are merged and ghost stubs are
cleaned up. Alert if it grows by >5% week-over-week — that means a
new pattern of unresolvable edges is appearing.

## What is NOT prevention

These are good ideas but don't belong in the prevention story:

- **Renaming every politician profile to a canonical name folder.** Huge
  editorial undertaking, doesn't pay back the cost. The librarian's
  aliasing makes the file-name choice irrelevant for read-path
  correctness. Different sessions can canonicalize incrementally.
- **Schema migration on `entities.jsonl`.** ADR-0023 covers schema
  enforcement separately. The prevention layer here is read-path
  unification, not write-path schema policing.
- **Forbidding wikilinks under non-canonical names.** The whole point
  of ADR-0024 is that the engine handles name ambiguity; the editor
  shouldn't have to remember which name form is "right" when writing.

## Sequencing recommendation

The three open tracks rank by payoff-per-session:

1. **Cache rebuilder migration (ADR-0024 Phase 3)** — eliminates Layer-1
   drift. ~1 session. Includes prerequisite: TS port of
   `edge-role-taxonomy.cjs` (~½ session of which can be done first if
   David wants to split it out).
2. **Four harness checks (this doc, Layer 3)** — closes the
   data-creation-drift gap. ~1 session for all four; they're small.
3. **Editorial duplicate-profile audit + decisions** — produces the
   list of "two folders for one human" cases, David picks canonicals.
   Audit script: ~30 min Code Claude. Editorial decisions: David's
   schedule.

No hard dependency between #1 and #2; they can ship in either order.
#3 surfaces work for David and is the only one of the three that
requires editorial bandwidth, not Code Claude bandwidth.

## Closing the prevention question

**The honest answer** to "how does this not become a problem in the
future" is: it stops becoming a problem when both Layer 1 (cache
through librarian) and Layer 3 (the four harness checks) are live.
Today we have Layer 2 fully working and Layer 1 partially working.
We caught instances of the bug class today; we haven't yet built the
trap that catches the next instance when it's created.

This document is the to-do list for closing that gap.
